# 技术博客翻译

## 目标

将英文技术写作翻译成忠实原文、自然易读、适合发布或整理笔记的简体中文。保留作者的含义、结构、细节层级和语气。除非用户明确要求，不要总结、扩写、评论或“优化”原文。

## 翻译原则

- 贴合原文含义翻译，而不是逐词硬译。保留每一个事实陈述、条件、限制、提醒和示例。
- 让中文表达自然、顺畅、易读，但不要新增解释、观点、转场、示例或结论。
- 保留作者语气：简洁的文档仍然简洁，口语化的博客仍然口语化，正式的参考文档仍然正式。
- 当技术精确性和文风润色冲突时，优先保证技术精确性。
- 不要因为句子看起来重复或显而易见就省略。原文重复强调的内容，译文也应体现这种重复。
- 不要弱化 `must`、`should`、`required`、`deprecated`、`experimental`、`warning`、`breaking change` 等表达的约束强度。

## 术语处理

- 产品名、模型名、库名、包名、API 名、协议名、文件名、路径、命令名、环境变量、类名、函数名、参数名和标识符保持英文。
- 常见技术术语优先使用通行译法，例如将 `context window` 译为“上下文窗口”，将 `prompt` 译为“提示词”，前提是符合上下文。
- 对不常见或有多种译法的术语，首次出现时使用 `中文（English）`，之后保持同一译法。
- API、SDK、CLI、HTTP、JSON、YAML、DOM、AST、LLM、RAG、GPU 等缩写保持不变，除非原文展开解释。
- 保留版本号、日期、基准测试名称、模型字符串、URL 和精确配置值。

## Markdown 与结构

- 尽量完整保留 Markdown 结构：标题、标题层级、列表、表格、引用块、提示块、代码围栏、链接、图片、脚注和 frontmatter。
- 翻译标题文本、段落文本、表格中的说明性文字、列表文字、引用块文字、图片 alt 文本，以及 note/warning 等提示块正文。
- 不要翻译 URL、链接目标、锚点 ID、文件路径、图片路径、代码标识符或 Markdown 引用标签，除非它们本身就是普通自然语言。
- 代码块保持不变。只有在用户明确要求时，或代码块明显是说明文字而非可执行代码时，才翻译其中内容。
- 行内代码保持不变，包括命令参数、配置键、对象字段、字面量和模型名。
- 保留表格列数和行顺序。只翻译面向读者的人类可读文本。
- 保留 frontmatter 的键名。只有 `title`、`description` 等面向读者的文本值可以翻译；slug、tag、date、path、ID 等保持不变。

## 工作流程

1. 判断源内容格式和输出目标：直接回复、新建文件，或原地编辑文件。
2. 翻译前阅读足够的上下文，先理解术语和主题。
3. 按章节逐段翻译，同时保留 Markdown 和原文顺序。
4. 在全文中保持术语译法一致。
5. 翻译完成后对照原文自检：
   - 每个标题和段落都已对应翻译。
   - 列表和表格仍保持原有结构。
   - 代码块、行内代码、链接、图片路径和标识符没有被误改。
   - 技术要求、限制条件和注意事项保留了原有强度。
   - 中文读起来自然，但没有加入原文没有的内容。
6. 如果某个短语确实有歧义，应根据上下文选择最可能的技术含义。只有当歧义会明显影响正确性时，才添加简短译者注；除非用户要求，否则优先把译者注放在译文正文之外。

## 输出规则

- 如果用户要求翻译文件并给出输出路径，将翻译后的 Markdown 写入该路径。
- 如果用户要求翻译文件但没有给出输出路径，默认创建同级文件，并使用清晰后缀，例如 `.zh.md`；如果项目已有命名约定，则遵循项目约定。
- 如果用户只要求在回复中给出翻译，除非用户要求说明，否则只返回译文。
- 默认不要附加摘要、术语表或译者注。
- 只有当保留策略或未解决的歧义确实重要时，才在译文后简短说明。

## 质量标准

最终译文应该像原技术文章的严谨中文版：内容忠实、术语准确、阅读顺畅，并且没有额外发挥。
