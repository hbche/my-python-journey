# 构建高效智能体 (Building Effective Agents)

**发布日期：** 2024 年 12 月 19 日

**作者：** Erik S. 和 Barry Zhang

在过去的一年中，我们与各行各业开发大语言模型（LLM）智能体的数十个团队进行了合作。我们发现，最成功的实现始终不是使用复杂的框架或专门的库，而是构建在简单、可组合的模式之上。

在本文中，我们将分享从客户合作及自身构建智能体的经验中总结出的成果，并为开发者提供构建高效智能体的实用建议。

---

## 什么是智能体？

“智能体”（Agent）可以通过多种方式进行定义。一些客户将智能体定义为完全自主的系统，这些系统可以在较长时间内独立运行，使用各种工具来完成复杂的任务。另一些人则使用该术语来描述遵循预定义工作流的更具指令性的实现。在 Anthropic，我们将所有这些变体归类为**智能体系统（agentic systems）**，但在工作流（workflows）**和**智能体（agents）之间做出了一个重要的架构区分：

- **工作流（Workflows）：** 是通过预定义的代码路径来编排 LLM 和工具的系统。
- **智能体（Agents）：** 则是 LLM 动态指导自身流程和工具使用，并保持对如何完成任务的控制的系统。

接下来，我们将详细探讨这两种类型的智能体系统。在附录 1（“智能体实践”）中，我们描述了客户在哪些领域发现了使用此类系统的独特价值。

---

## 何时（以及何时不）使用智能体

在使用 LLM 构建应用程序时，我们建议寻找尽可能简单的解决方案，并且仅在需要时才增加复杂度。这可能意味着根本不需要构建智能体系统。智能体系统往往以延迟和成本为代价来换取更好的任务表现，您应该考虑这种权衡在什么时候是有意义的。

当确实需要更高的复杂度时，**工作流**为定义明确的任务提供了可预测性和一致性；而当在大规模场景下需要灵活性和模型驱动的决策时，**智能体**则是更好的选择。然而，对于许多应用来说，通过检索（Retrieval）和上下文示例（In-context examples）来优化单次 LLM 调用通常就已经足够了。

---

## 何时以及如何使用框架

有许多框架可以让智能体系统更容易实现，包括：

- Claude Agent SDK
- AWS 的 Strands Agents SDK
- Rivet（一个拖拽式 GUI LLM 工作流构建器）
- Vellum（另一个用于构建和测试复杂工作流的 GUI 工具）

这些框架通过简化标准的底层任务（如调用 LLM、定义和解析工具以及将调用链连接在一起）让上手变得轻松。然而，它们通常会创建额外的抽象层，从而可能掩盖底层的提示词（prompts）和响应（responses），使调试变得更加困难。它们还可能诱使开发者在原本简单的设置就足够时去增加系统的复杂度。

我们建议开发者先直接使用 LLM API：许多模式只需几行代码即可实现。如果您确实使用了某种框架，请务必理解其底层代码。对“引擎盖下”发生的事情产生错误假设是客户出错的常见原因。

可以参考我们的 Cookbook 以获取一些示例实现。

## 基础构建块、工作流与智能体

在本节中，我们将探讨我们在生产环境中见过的智能体系统的常见模式。我们将从我们的基础构建块——**增强型 LLM**——开始，并逐步增加复杂度，从简单的组合工作流到自主智能体。

### 基础构建块：增强型 LLM (The Augmented LLM)

智能体系统的基本构建块是经过增强的 LLM，这些增强包括**检索（retrieval）**、**工具（tools）**和**记忆（memory）**。我们目前的模型可以主动利用这些能力——生成自己的搜索查询、选择合适的工具并确定要保留哪些信息。

我们建议在实现时重点关注两个关键方面：针对您的特定用例定制这些能力，并确保它们为您的 LLM 提供简单且文档完善的接口。虽然有许多方法来实现这些增强，但其中一种途径是通过我们最近发布的**模型上下文协议（Model Context Protocol, MCP）**，它允许开发者通过简单的客户端实现与日益壮大的第三方工具生态系统进行集成。

在本文的其余部分中，我们将假设每次 LLM 调用都可以使用这些增强能力。

---

### 工作流：提示词链 (Prompt Chaining)

提示词链将一项任务分解为一系列步骤，其中每次 LLM 调用都会处理前一次调用的输出。您可以在任何中间步骤中添加程序化检查（例如流程中的“网关/闸口 (gate)”），以确保流程仍处于正常轨道。

- **何时使用此工作流：** 此工作流非常适合可以轻松且干净地分解为固定子任务的情况。其主要目标是通过使每次 LLM 调用承担更简单的任务，从而以延迟为代价换取更高的准确性。
- **适用示例：**

1. 生成营销文案，然后将其翻译成另一种语言。
2. 撰写文档大纲，检查大纲是否符合某些标准，然后根据大纲撰写整篇文档。

---

### 工作流：路由 (Routing)

路由对输入进行分类，并将其定向到专门的后续任务。这种工作流实现了关注点分离，并允许构建更具专业性的提示词。如果没有这种工作流，针对某一种输入进行优化可能会损害系统在其他输入上的表现。

- **何时使用此工作流：** 路由非常适用于包含不同类别的复杂任务，这些类别最好分开处理，并且分类可以由 LLM 或更传统的分类模型/算法准确处理。
- **适用示例：**

1. 将不同类型的客户服务查询（常规问题、退款请求、技术支持）分流到不同的下游流程、提示词和工具中。
2. 将简单/常见的问题路由到更小、成本效益更高的模型（如 Claude Haiku 4.5），而将困难/罕见的问题路由到能力更强的模型（如 Claude Sonnet 4.5），以平衡成本并优化最佳性能。

---

### 工作流：并行化 (Parallelization)

LLM 有时可以同时处理一项任务，并由程序自动聚合其输出。这种并行化工作流主要表现为两种形式：

1. **分块（Sectioning）：** 将任务分解为并行运行的独立子任务。
2. **投票（Voting）：** 多次运行同一任务以获得多样化的输出。

- **何时使用此工作流：** 当分解后的子任务可以通过并行化来提高速度，或者需要多个视角或多次尝试以获得更高置信度的结果时，并行化非常有效。对于包含多个考量的复杂任务，当每个考量都由单独的 LLM 调用处理时，LLM 通常表现得更好，因为这使它能够专注于每个特定方面。
- **适用示例：**
- **分块（Sectioning）：**
- 实施护栏（Guardrails），其中一个模型实例处理用户查询，而另一个实例则审查这些查询中是否包含不当内容或请求。这往往比使用同一次 LLM 调用同时处理护栏和核心响应表现得更好。
- 自动化评估（Evals）以评估 LLM 的表现，其中每次 LLM 调用都会从不同方面评估模型在给定提示词下的表现。

- **投票（Voting）：**
- 审查某段代码是否存在漏洞，其中几个不同的提示词进行独立审查，并在发现问题时对代码进行标记。
- 评估某段给定的内容是否不当，通过多个提示词评估不同的方面，或者设置不同的投票阈值以平衡误报（false positives）和漏报（false negatives）。

---

### 工作流：编排者-工作者 (Orchestrator-Workers)

在“编排者-工作者”工作流中，一个中央 LLM 动态地分解任务，将其委派给多个工作者（Worker）LLM，然后对它们的结果进行综合。

- **何时使用此工作流：** 此工作流非常适合无法预测所需子任务的复杂任务（例如，在编程中，需要修改的文件数量以及每个文件中修改的性质很可能完全取决于具体任务）。尽管它在结构上与并行化相似，但关键区别在于其**灵活性**——子任务并不是预先定义的，而是由编排者根据具体的输入内容动态决定的。
- **适用示例：**

1. 代码开发产品：每次需要对多个文件进行复杂的连锁修改。
2. 复杂搜索任务：涉及从多个来源收集和分析信息以获取潜在相关信息。

---

### 工作流：评估者-优化者 (Evaluator-Optimizer)

在“评估者-优化者”工作流中，一个 LLM 调用负责生成响应，而另一个调用则在循环中提供评估和反馈。

- **何时使用此工作流：** 当我们有明确的评估标准，且迭代微调能带来可衡量的价值时，这种工作流特别有效。适合采用此模式的两个迹象是：第一，当人类明确表达反馈时，LLM 的响应能够得到明显的提升；第二，LLM 本身有能力提供这种有效的反馈。这类似于人类作家在创作一篇精雕细琢的文档时所经历的迭代写作过程。
- **适用示例：**

1. 文学翻译：其中可能存在翻译者 LLM 最初未能捕捉到的细微差别，而评估者 LLM 可以提供有用的批评和改进意见。
2. 复杂搜索任务：需要多轮搜索和分析以收集全面信息，其中评估者将决定是否有必要进行进一步的搜索。

---

## 自主智能体 (Autonomous Agents)

随着 LLM 在核心能力（理解复杂输入、进行推理与规划、可靠地使用工具以及从错误中恢复）上的日益成熟，智能体（Agents）正在生产环境中脱颖而出。

智能体的工作始于人类用户的指令或与用户的交互式讨论。一旦任务明确，智能体就会独立进行规划和操作，并可能在需要更多信息或主管判断时重新求助于人类。

在执行过程中，智能体在每一步都从环境中获取“地面实况”（ground truth）（例如工具调用结果或代码执行结果）以评估其进度，这是至关重要的。智能体随后可以在检查点或遇到阻碍时暂停以等待人类的反馈。任务通常在完成时终止，但为了保持控制，加入终止条件（例如最大迭代次数）也很常见。

智能体可以处理复杂的任务，但它们的实现往往非常直接。它们通常只是 LLM 在循环中根据环境反馈来调用工具。因此，清晰且周密地设计工具集及其文档至关重要。我们将在附录 2（“针对工具进行提示词工程”）中详细介绍工具开发的最佳实践。

- **何时使用智能体：** 智能体可用于开放式问题，在这些问题中，很难或不可能预测所需的步骤数量，并且您无法硬编码一条固定路径。LLM 可能会运行很多轮，您必须对其决策保持一定程度的信任。智能体的自主性使其成为在受信任环境中扩展任务规模的理想选择。
- **注意事项：** 智能体的自主特性也意味着更高的成本以及错误复合放大（compounding errors）的风险。我们建议在沙盒（sandboxed）环境中进行广泛的测试，并配备适当的护栏。
- **适用示例（来自 Anthropic 内部的实现）：**

1. 一个用于解决 SWE-bench 任务的编程智能体（Coding Agent），该任务涉及根据任务描述自动修改仓库中的许多文件。
2. 我们的“计算机使用”（computer use）参考实现，其中 Claude 可以像人类一样通过操作虚拟计算机屏幕来完成任务。

---

## 组合与定制这些模式

这些构建块并不是一成不变的指令。它们是常见模式，开发者可以根据不同用例进行改造和组合。

与任何 LLM 功能一样，成功的关键在于**衡量性能**并对实现进行**持续迭代**。重申一遍：只有当增加复杂度能够显著且可证明地改善结果时，才应该考虑增加复杂度。

---

## 总结

在 LLM 领域，成功不在于构建最复杂的系统，而在于构建最适合您需求的系统。从简单的提示词开始，通过全面的评估（Evals）来优化它们，并且只有在更简单的解决方案无法满足需求时，才加入多步骤的智能体系统。

在实现智能体时，我们尝试遵循以下三个核心原则：

1. **保持设计简单：** 维持智能体设计的简单性与纯粹性。
2. **保证高透明度：** 优先考虑透明度，明确展示智能体的规划与执行步骤。
3. **精心打磨接口：** 通过详尽的工具文档和测试，精心打造您的“智能体-计算机接口”（Agent-Computer Interface, ACI）。

框架可以帮助您快速上手，但在走向生产环境时，不要犹豫去减少抽象层并使用基础组件进行构建。通过遵循这些原则，您可以创建出不仅强大，而且可靠、易于维护且深受用户信赖的智能体。

## 附录 1：智能体实践

我们与客户合作的过程中，发现了 AI 智能体两个特别有前景的应用，它们展示了上述模式的实际价值。这两个应用都说明了智能体在以下任务中能带来最大价值：需要同时进行对话与行动、有明确的成功标准、支持反馈循环并集成了有效的人类监督。

### A. 客户支持 (Customer support)

客户支持将熟悉的聊天机器人界面与通过工具集成带来的增强能力相结合。这非常自然地契合了更具开放性的智能体，因为：

- 支持交互自然地遵循对话流，同时需要访问外部信息和执行操作；
- 可以集成工具来拉取客户数据、订单历史和知识库文章；
- 可以通过编程方式处理诸如发放退款或更新工单等操作；并且
- 成功可以通过用户定义的解决方案（例如问题是否解决）来进行清晰的衡量。

几家公司已经通过基于使用情况的定价模型（仅对成功的解决方案收费）证明了这种方法的可行性，这展示了它们对其智能体有效性的信心。

### B. 编程智能体 (Coding agents)

软件开发领域在 LLM 功能方面展现出了巨大的潜力，其能力已从代码补全演变为自主解决问题。智能体之所以特别有效，是因为：

- 代码解决方案可以通过自动化测试进行验证；
- 智能体可以使用测试结果作为反馈来对解决方案进行迭代；
- 问题空间定义明确且结构化；并且
- 输出质量可以客观地（通过测试通过率）进行衡量。

在我们自己的实现中，智能体现在仅凭 Pull Request 描述就能够解决 SWE-bench Verified 基准测试中的真实 GitHub 问题。然而，尽管自动化测试有助于验证功能，但人类审查（Code Review）对于确保解决方案符合更广泛的系统架构要求仍然至关重要。

## 附录 2：针对工具进行提示词工程 (Prompt engineering your tools)

无论您正在构建哪种智能体系统，工具都可能是您智能体的重要组成部分。工具通过在我们的 API 中指定其确切的结构和定义，使 Claude 能够与外部服务和 API 进行交互。当 Claude 做出响应时，如果它计划调用某个工具，它将在 API 响应中包含一个 `tool_use` 块。工具的定义和规范应该像整体提示词一样，得到同等程度的提示词工程关注。

指定同一个动作通常有几种方法。例如，您可以通过编写 diff（补丁）来指定文件修改，也可以通过重写整个文件来实现。对于结构化输出，您可以返回 Markdown 内部的代码，也可以返回 JSON 内部的代码。在软件工程中，这些差异只是表面形式上的，可以相互无损转换。然而，对于 LLM 而言，某些格式的编写难度要比其他格式大得多：

- **编写 diff** 要求在写入新代码之前，就已经在块头部（chunk header）精确知道有多少行正在发生变化。
- **在 JSON 内部编写代码**（与 Markdown 相比）需要对换行符和引号进行大量的额外转义。

我们在决定工具格式时的建议如下：

1. **给模型足够的 Token 在动笔前进行“思考”**，以免其陷入僵局（writes itself into a corner）。
2. **使格式接近模型在互联网文本中自然见到的形式。**
3. **确保没有格式上的“开销/超载”**，例如必须精确计算数千行代码，或者对自己编写的任何代码进行字符串转义。

一个经验法则是，想想我们在人机交互（HCI）上投入了多少精力，就应该计划在创建良好的智能体-计算机接口（Agent-Computer Interface, ACI）上投入同样多的精力。以下是关于如何实现这一一些心得：

- **设身处地为模型着想：** 根据描述和参数，该工具的使用方法是否显而易见，还是需要你仔细思考？如果是后者，那么对模型来说可能也是如此。一个好的工具定义通常包括使用示例、边缘情况、输入格式要求以及与其他工具的清晰界限。
- **像为初级开发人员写文档：** 思考如何更改参数名称或描述来让事情变得更易懂。把这看作是为你团队中的初级开发人员编写一份极佳的文档字符串（docstring）。当使用许多相似的工具时，这一点尤为重要。
- **测试模型如何使用您的工具：** 在我们的 Workbench 中运行许多示例输入，看看模型会犯什么错误，并进行迭代。
- **进行防错设计（Poka-yoke / 防呆）：** 修改参数和参数结构，使其更不容易被模型误用。

在为 SWE-bench 构建我们的智能体时，我们实际上在**优化工具**上花费的时间比在整体提示词上还要多。例如，我们发现，在智能体移出根目录后，模型在使用采用相对文件路径的工具时会频繁犯错。为了解决这个问题，我们修改了工具，要求其始终必须使用**绝对文件路径**——结果我们发现模型此后完美无瑕地使用了这种方法。
