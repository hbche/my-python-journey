# 提示工程（Prompt Engineering）

你好！既然你打算把 **Prompt Engineering** 真正作为日常开发和打造 Agent Skill 的生产力武器，那我们就摒弃那些流于表面的“咒语指南”。

今天，我将以**费曼学习法（Feynman Technique）**的核心精髓——**“用极其通俗的语言，把复杂的概念拆解给外行听，并在这个过程中直击本质”**——来为你深入复盘和详解 Prompt Engineering。

我们要拆解的不是一堆死板的模版，而是支撑这些模版背后的**核心底层逻辑**。

---

## 第一阶段：概念重构——什么是 Prompt 的本质？

用费曼学习法最经典的“大白话比喻”来解释：

> **传统编程**就像是给机器喂**汇编/高级语言代码**，你需要精确到每一个寄存器和内存指针；
> **Prompt Engineering** 则像是给一个**全知全能但极度缺乏常识、有些健忘的顶级实习生**写**《工作说明书（SOW）》**。

很多人以为 Prompt 只是“提问”，但在资深 Prompt Engineer 眼里，Prompt 的本质是“边界设定（Boundary Setting）”**和**“上下文路由（Context Routing）”。

大模型（LLM）是一个拥有数千亿参数的概率空间。如果你问得太宽泛，它就会在庞大的概率空间里随机游走，给出平庸的回答。**Prompt Engineering 的核心目的，就是通过精准的文本输入，在 LLM 的知识库里“圈出一块特定领地”，强迫它在这个特定的逻辑轨道上运行。**

---

## 第二阶段：核心机制拆解（深入底层）

为了让你在写 Claude Code 或 Codex 的 Skill 时能够“一把过”，我们需要把 Prompt 拆解为四个最核心的动力组件。我们用“组装一台精密自动化机器”来对应解释：

### 1. 角色与设定（Role & Meta-Instruction）

- **费曼大白话**：告诉 AI 它是谁，以及它**不能**是谁。
- **深度解析**：在 Transformer 架构中，前置的角色设定会直接影响后续 Token（字符）生成的概率分布。如果你指定 `You are a Senior Frontend Architect`，模型会立刻提高与架构设计、设计模式、性能优化相关词汇的激活概率。
- **针对 Agent Skill 的高阶技巧**：不要只写“你是个资深专家”。要给它**限制性人格**。例如：
  > “你是一个极其挑剔、严格遵守 KISS 原则（Keep It Simple, Stupid）的资深架构师。你对冗余代码和过度设计有生理性的反感。”
  > 这样能大幅收敛 Agent 的发散思维，让输出极度聚焦。

### 2. 上下文与约束（Context & Constraints）

- **费曼大白话**：划定赛道，安上护栏，明确告诉它“什么能做，什么绝对不能碰”。
- **深度解析**：Agent（如 Claude Code）在执行任务时，最容易犯的错误就是“越界”——自己脑补需求，或者修改了不该修改的文件。**约束条件是 Skill 稳定性的生命线。**
- **经典三元约束法**：
  - **输入限制**：明确期望的数据结构（如：只接受特定 Monorepo 目录下的 `package.json`）。
  - **过程限制**：规定思考路径（如：在修改任何代码前，必须先在 `scratchpad` 中列出受影响的依赖）。
  - **输出限制**：严格限定格式（如：只能输出标准的 JSON，禁止任何解释性文本）。

### 3. 示例引导（Few-Shot Prompting / In-Context Learning）

- **费曼大白话**：千言万语的规则，不如直接塞给它两个“满分作文”的范例。
- **深度解析**：大模型具备极强的“上下文学习”能力。给出的示例不仅是让它模仿格式，更重要的是让它模仿**推理逻辑的严密程度**。
- > **注意**：在为 Codex 或 Claude Code 编写自动化 Skill 时，示例必须包含**“坏用例 -> 思考过程 -> 好用例”**的对比。让 Agent 明白什么是“拒绝跨越的红线”。

### 4. 思维链与工程化路径（Chain of Thought, CoT）

- **费曼大白话**：逼着 AI 在给出最终答案前，把“草稿纸”传给你看，写出“因为、所以”的推导步骤。
- **深度解析**：数学题或复杂的架构重构不能直接要答案。通过加入 `Let's think step by step`（让我们一步步思考），或者在 Skill 中显式设计 `<thinking>` 标签，是在强迫模型在自回归生成过程中，利用前文的 Token 作为后续推理的“工作内存（Working Memory）”。这能将复杂逻辑任务的准确率提升数倍。

## 第三阶段：费曼实战模拟——设计一个“高稳定、有边界”的 Agent Skill

现在，我们把上述底层逻辑具象化。假设我们要为 Claude Code 封装一个日常高频任务：**自动化国际化（i18n）文案提取与词条库比对 Skill**。

如果我们写粗漏的 Prompt，Agent 很容易漏掉字符串，或者把代码结构改崩。下面我们用“深入且严密”的工程化思维来写这个 Prompt：

```markdown
# Role

你是一个自动化的前端国际化（i18n）代码重构机器人。你唯一的任务是扫描文件，提取硬编码的中文，并将其替换为 i18n 变量。

# Operational Pipeline (思维链控制)

当你接收到代码文件时，必须严格按照以下三步执行，严禁跳过：

1. <analysis>:
   - 逐行扫描代码，列出所有硬编码的中文文本。
   - 检查企业统一词条库中是否已存在该中文的 key。如果存在，记录该 key；如果不存在，按照 `module.page.element` 的规范拟定一个新的 key。
2. <verification>:
   - 检查这些中文是否属于“不需要翻译的硬编码”（例如：console.log、注释、内部错误码）。如果是，将其剔除出重构列表。
3. <transformation>:
   - 仅执行替换操作。保持原有代码的逻辑、缩进和上下文完全一致。

# Constraints (硬性边界)

- 🚫 【绝对禁止】修改任何业务逻辑、变量命名或组件生命周期。
- 🚫 【绝对禁止】在没有完全把握的情况下猜测未定义的变量。
- 🎯 【代码安全】如果单行代码中混杂了复杂的 JavaScript 模板字符串（如 `${}`），必须优先在 `<analysis>` 阶段拆解伪代码，确认无误后再替换。
- 📝 【输出规范】最终只允许输出重构后的完整代码文件，禁止包含任何“好的，我已经为您修改了代码”等礼貌性客套话。

# Examples

(此处应放入一段包含复杂中文嵌套的代码替换前后的标准范例...)
```

## 第四阶段：反思与桥接——如何让你的 Skill “稳定、可靠、不绕弯路”？

既然你在实际工作中经常需要封装 Skill，结合费曼学习法的“查漏补缺（Review and Simplify）”，资深 Prompt Engineer 给你的**避坑金律**只有三条：

1. **用“声明式”代替“祈使句”**：

- ❌ 别说：“请你尽量不要修改其他文件。”（AI 会认为这是建议，从而可能违反）。
- Target 🎯 说：“**当前操作的上下文域（Scope）严格限制在当前激活的文件内。跨文件修改将被视为严重违规。**”

2. **设计“熔断机制（Kill Switch）”**：

- 在写复杂的自动化脚本时，给 Prompt 加一把锁：“**如果在 `<analysis>` 阶段发现硬编码中文超过 50 处，或者遇到无法解析的 AST 语法糖，立即输出 `[ERROR: EXCEEDS_LIMIT]` 并终止退出，严禁擅自修改。**” 这样可以防止 AI 把你的整个项目搞乱。

3. **把输入和输出结构化**：

- 多利用 Markdown 的标题（`#`）、列表（`-`）和 XML 标签（`<thinking>`、`<output>`）。大模型对结构化标记的注意力权重，远高于大段的纯文本叙述。

## 费曼总结

Prompt Engineering 绝不是一门“玄学”，它是**用确定性的结构去规范模糊的自然语言，从而驾驭不确定性的概率模型**。

在你编写 Claude Code 或 Codex 的 Skill 时，始终问自己三个费曼式的问题：

1. 我把任务的目标和边界说得足够直白、连实习生都能听懂吗？
2. 我有没有为它安上“绝对不能做”的护栏？
3. 它在动手前，有地方让它写“草稿（Thinking 过程）”吗？

你在日常编写 Skill 的过程中，目前遇到最让你头疼、或者 AI 最容易“走偏”的具体场景是什么？我们可以针对那个具体场景，现场重构一个高抗噪、极度稳定的 Skill Prompt。
