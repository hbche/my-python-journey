# 提示词链(Prompt Chain)

前面我们已经理解了 Prompt 的本质（即**对大模型潜空间状态的引导与约束**），那么我们就可以直接跳过那些基础的“套路”和“模板”，进入到真正能解决复杂业务问题的核心技术——**Prompt Chain（提示词链）**。

在资深 Prompt 工程师的视角里，单一的 Prompt 就像是单步指令，它的上限受限于模型的上下文推理窗口和单次输出的注意力集中度。而 **Prompt Chain，则是将一个复杂的任务拆解为多个子任务，并将前一步的输出作为后一步的输入，串联成一个有向无环图（DAG）或线性工作流。**

以下是我为你梳理的 Prompt Chain 核心方法论，帮助你从“写提示词的人”蜕变为“设计大模型工作流的架构师”。

---

## 1. 为什么要用 Prompt Chain？（核心痛点）

在面对复杂任务（如：写一篇深度行业报告、分析一份复杂的法律合同、开发一个多步骤的代码重构工具）时，单次 Prompt 往往会遇到以下瓶颈：

- **注意力涣散（Lost in the Middle）：** 模型在单次生成中，很难同时兼顾逻辑结构、事实准确性、代码规范和语气风格。
- **缺乏中间思考：** 就像人类无法在一秒内凭空解出复杂的微积分方程一样，大模型也需要“草稿纸”和“中间步骤”。
- **容错率极低：** 如果单次生成的中间某个环节出错，整个输出就会崩塌，且无法局部纠错。

**Prompt Chain 的解法：** **分而治之（Divide and Conquer）**。通过确定性的工程化流程，降低模型在每一步的熵（不确定性），从而极大地提升复杂任务的成功率。

---

## 2. Prompt Chain 的三种典型拓扑结构

在设计链式提示词时，我们通常会根据业务逻辑设计不同的拓扑结构：

### 线性链（Linear Chain）

最基础的结构。上一步的输出完全作为下一步的输入。

> **数据流：** 输入 -> [步骤 A：提取核心观点] -> 中间结果 -> [步骤 B：扩写段落] -> 最终输出。

### 并行聚合链（Parallel-Aggregation Chain）

将输入分发给多个并行的 Prompt 处理，最后由一个聚合 Prompt 进行汇总。这在处理海量文本或多维度分析时非常有用。

> **数据流：**
>
> - 输入 -> [步骤 A1：提取技术痛点] -> 结果 1 \
> - 输入 -> [步骤 A2：提取商业价值] -> 结果 2 --> [步骤 B：整合商业计划书] -> 最终输出
> - 输入 -> [步骤 A3：提取竞争对手] -> 结果 3 /

### 路由/分支链（Routing Chain）

引入一个“分类器（Router）”判断输入的意图，然后根据判断结果跳转到不同的专业 Prompt 链中。

> **数据流：** 输入 -> [Router：判断是 Bug 还是新需求]
>
> - 如果是 Bug -> 进入 [Bug 修复链]
> - 如果是新需求 -> 进入 [功能设计链]

## 3. 资深工程师的实战拆解：以“技术博客自动生成”为例

假设我们要让模型写一篇高质量的技术博客。单次 Prompt 往往会生成一堆流于表面的套话。我们将其拆解为 3 步构成的 Prompt Chain：

### 🧱 节点 1：大纲生成与逻辑校验（Chain Node 1）

- **输入：** 原始的技术文档或核心概念。
- **Prompt 核心逻辑：** 专注于结构设计，强制模型进行逻辑自洽性检查，输出结构化的 JSON 大纲。
- **输出：** `outline.json`

### 🧱 节点 2：分段深度扩写（Chain Node 2）

- **输入：** `outline.json` 中的某一个子章节 + 上下文背景。
- **Prompt 核心逻辑：** 此时模型**不需要**考虑全局结构，只需要专注于当前的这一段。要求其补充代码示例、使用具体的比喻，并保持技术严谨性。
- **输出：** `raw_content_chapter_X.md`

### 🧱 节点 3：润色、代码审查与格式化（Chain Node 3）

- **输入：** 拼接后的 `raw_content.md`。
- **Prompt 核心逻辑：** 此时模型转变为“编辑”和“Linter”角色。专门检查代码是否能跑通、文风是否统一、 Markdown 语法是否正确。
- **输出：** 最终发布版本。

## 4. Prompt Chain 的核心设计原则（避坑指南）

作为一名严谨的工程师，在设计 Prompt Chain 时，你需要严格遵守以下原则：

1. **严格控制输入输出协议（I/O Protocol）：**
   在链条的中间节点，**强烈建议使用 JSON 或 XML 作为交互媒介**。不要让模型输出纯自然语言供下一步解析。例如，在 Prompt 尾部加上：

```markdown
Return ONLY a JSON object matching this schema: {"key": "value"}. Do not include any conversational text.
```

2. **单一职责原则（Single Responsibility）：**
   每一个 Prompt 节点只做一件事。如果一个节点既要“翻译”又要“总结”还要“转换格式”，请立刻把它拆成三个节点。
3. **状态隔离与上下文裁剪：**
   不要把前面所有步骤的所有原始信息都一股脑传给下一步。这会导致模型注意力分散。下一步只需要**上一步精炼后的结果**以及**必要的全局上下文**。

---

## 5. 从 Prompt Chain 到 Agent 的演进

当你掌握了 Prompt Chain，你会发现它的局限性：流程是死板的、固定的。如果中间某一步模型彻底理解错了，程序还是会硬着头皮往下走。

为了解决这个问题，你可以在链条中引入“评估节点（Evaluation Node）”，即让大模型自己扮演测试官，审查上一步的输出是否合格。如果不合格，通过 Loop（循环）打回重做。这就演进成了 **Self-Correction（自省机制）**，也是迈向 AI Agent（智能体）的核心一步。

## 6. Prompt Chain 不是一段 Prompt

**Prompt Chain（提示词链）** 的核心本质是**将一个复杂的任务拆解为多个子任务，并将多个不同的 Prompt 按照逻辑顺序、“分批次”地喂给大模型（LLM）**。

在实际运行中，**前一个 Prompt 的输出，通常会作为下一个 Prompt 的输入（或输入的一部分）**。

为了让你更清晰地理解，我们可以对比一下你提到的这两种模式：

---

### 6.1. 什么是真正的 Prompt Chain？（分多段 Prompt 分批次喂给 LLM）

这就是标准的 Prompt Chain。它类似于工厂的**流水线（Pipeline）**。大模型不是一次性完成所有事情，而是走一步、看一步，每一步都专注做好一件小事。

#### 💡 真实场景示例：写一篇行业分析报告

如果不采用 Chain，你必须写一个极其复杂的 Prompt，让大模型同时注意格式、逻辑、数据和文风，大模型很容易“顾此失彼”（幻觉或漏掉关键信息）。

如果使用 **Prompt Chain**，流程会被拆解为以下批次：

```
[用户初始输入: 主题 "新能源汽车"]
       │
       ▼
【Prompt 1: 提纲生成】 🔍 "请针对主旨生成一份详细的报告大纲..."
       │
       ├─► 得到输出：$大纲$
       │
       ▼
【Prompt 2: 资料扩充】 📚 "基于以下$大纲$，请为每个章节列出核心知识点和所需数据..."
       │
       ├─► 得到输出：$丰富后的素材$
       │
       ▼
【Prompt 3: 正文撰写】 ✍️ "请根据$丰富后的素材$，用专业、严谨的文风撰写最终报告..."
       │
       └─► 得到最终完美的报告

```

**为什么这样做效果更好？**

- **降低认知负荷**：大模型和人一样，单次任务越单一，注意力越集中，准确率越高。
- **突破 Context 限制**：中间步骤可以引入外部工具（如联网搜索、数据库查询）来补充信息，再喂给下一个 Prompt。
- **便于调试**：如果最终报告写得不好，你可以一眼看出是“大纲生成（Step 1）”出了问题，还是“正文撰写（Step 3）”出了问题，从而精准优化某一段 Prompt。

---

### 6.2. 什么是“一段 Prompt 分流程完成”？（Single Prompt with Multi-steps）

如果你在**同一个 Prompt** 里面写：“第一步，请先思考...；第二步，请列出提纲...；第三步，根据提纲输出回答...”，这在技术上通常被称为 **思维链（CoT, Chain of Thought）** 或 **结构化提示词**，而不是 Prompt Chain。

- **特点**：只有一次 I/O（一次输入，一次输出）。
- **劣势**：大模型在生成前面的步骤时，如果出错（Step 1 跑偏了），后面的步骤会自动沿着错误的道路走下去，你作为开发者在中间**无法干预、无法拦截、也无法修正**。

---

### 🛠️ 总结与开发落地

- **Prompt Chain 必定是“多轮交互”或“多段分批”的**。
- 在实际工程开发中（例如使用 LangChain、Vercel AI SDK 或自己写 Python/TS 脚本），你通常会写一个 `for` 循环或 `async/await` 异步流，把上一个 `LLMResponse` 塞进下一个 `LLMRequest` 的 `messages` 数组里。
