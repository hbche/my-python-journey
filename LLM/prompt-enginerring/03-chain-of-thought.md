# 思维链(Chain of Thought)

在 Prompt Chain 的体系中，**CoT (Chain-of-Thought，思维链)** 是一个极其核心的飞跃。如果说 Prompt Chain 是在宏观上**由你（人类）来规划**任务的解构步骤，那么 CoT 就是在微观上**引导大模型自己**进行深度的“自省”和分步推理。

下面我们来深度拆解 CoT 的核心本质、运作原理以及高级工程实践。

---

## 一、 什么是 CoT（思维链）？

**CoT 的本质，是强制大模型将“直觉式快思考”转化为“理性的一步步慢思考”。**

- **传统 Prompt（Direct Prompting）**：直接向模型要答案。模型通过概率预测下一个 Token，试图一步到位。面对复杂逻辑时，极易由于计算路径过短而产生“幻觉”或计算错误。
- **CoT Prompt**：在给出答案之前，强制输出一段**连续的、有逻辑的中间推理步骤**。

例如，如下的Prompt Engineering 示例：

提示：

```txt
我去市场买了10个苹果。我给了邻居2个苹果和修理工2个苹果。然后我去买了5个苹果并吃了1个。我还剩下多少苹果？
```

输出：

```txt
11个苹果
```

答案是不对的，现在我们来尝试使用**特殊提示**。

提示：

```txt
我去市场买了10个苹果。我给了邻居2个苹果和修理工2个苹果。然后我去买了5个苹果并吃了1个。我还剩下多少苹果？

让我们逐步思考。
```

输出：

```txt
首先，您从10个苹果开始。
您给了邻居和修理工各2个苹果，所以您还剩下6个苹果。
然后您买了5个苹果，所以现在您有11个苹果。
最后，您吃了1个苹果，所以您还剩下10个苹果。
```

令人印象深刻的是，这个简单的提示在这个任务中非常有效。这在我们没有太多示例可用于提示时特别有用。

### 为什么 CoT 能有效提高模型准确率？

大模型是基于 Token 逐字生成的。当它直接输出答案时，它只有一次计算机会。而当它输出推理步骤时，**前一步生成的文本会作为上下文（Context）输入给下一步**。这相当于给大模型提供了一个“草稿纸”，让它能够“看着自己前面的思考过程”来推导下一步，从而极大地扩充了模型的有效计算量。

### 核心本质

大模型生成文本的逻辑是：根据当前输入的所有 Token（词元），预测下一个最可能出现的Token。

CoT 的本质是利用模型自主生成的中间 Token 作为“逻辑垫脚石”。它把一个复杂的、需要一步到位的高难度计算，拆解成了多个连续的、概率极高的低难度条件概率预测。

## 二、 CoT 的两种经典范式

在实际工程中，CoT 主要有两种实现方式：

### 1. Zero-shot CoT（零样本思维链）

这是由 Kojima 等人在 2022 年发现的“魔法”。你不需要给模型任何例子，只需要在 Prompt 的结尾加上一句充满魔力的咒语：

> **"Let's think step by step."（让我们一步一步地思考。）**

不要小看这句话。在数学、逻辑推理任务中，加上这句话往往能让基础模型的准确率实现断崖式提升。它**激活了模型在预训练阶段吸收的流式推理数据**。

### 2. Few-shot CoT（少样本思维链）

这是由 Wei 等人在 2022 年提出的正统 CoT。通过向模型提供 **「问题 -> 思考过程 -> 答案」** 的完整示例（Demonstration），教导模型模仿这种推理格式。

#### 💡 资深工程师案例对比：

- **糟糕的 Few-shot（普通示例）**：

  > **Q**: 餐厅里有 23 个苹果。如果用了 20 个做午餐，又买了 6 个，现在有多少个苹果？
  > **A**: 9 个。

- **高效的 Few-shot CoT（思维链示例）**：
  > **Q**: 餐厅里有 23 个苹果。如果用了 20 个做午餐，又买了 6 个，现在有多少个苹果？
  > **A**: 餐厅原本有 23 个苹果。用了 20 个做午餐，还剩 23 - 20 = 3 个。接着又买了 6 个，所以现在有 3 + 6 = 9 个。答案是 9。

第二种示例中在回答中，给出了详细的计算细节和步骤，也是在告诉大模型，后续提问，你也要想这样详细地按步骤推理计算。

#### Few-shot 的本质

Few-shot 的本质是激活模型的“模仿本能”。

大模型极度依赖模式匹配

- 如果我们的 Few-shot 示例按照上一结中示例1给出，在回答中直接给出答案，这就是在告诉模型一个潜规则：“在这个当前的会话中，我们的对话风格是极其简短的，请直接给我最终结果。” 模型为了迎合这种风格模仿，就会强行闭眼盲猜一个答案。
- 如果我们的 Few-shot 示例里给的是带有推理过程的 CoT，你实际上是在向模型演示一种“思维范式”。模型通过上下文学习（In-context Learning），精准捕捉到了这种“先解释、再计算、最后出答案”的文本结构，从而在处理接下来的未知新问题（比如更复杂的数学题）时，也会强制自己模仿这种分步输出的格式。

## 三、 CoT 与 Prompt Chain 的辩证关系

既然你提到了 Prompt Chain，我们必须厘清这两者的边界与融合。**它们不是替代关系，而是互补关系。**

| 维度         | Prompt Chain (提示词链)                                  | CoT (思维链)                                         |
| ------------ | -------------------------------------------------------- | ---------------------------------------------------- |
| **控制主体** | **人类工程师**。你规划好 Step 1 算什么，Step 2 传什么。  | **大模型自身**。由模型在单次调用内自主决定推理细节。 |
| **颗粒度**   | **宏观级**。跨越多个 API 请求/交互模块。                 | **微观级**。存在于单次 API 请求的 Token 生成过程中。 |
| **应用场景** | 复杂的工作流、需要外部工具接入、需要严格合规的业务逻辑。 | 复杂的计算、符号推理、幽默理解、常识推理。           |

### 🚀 进阶玩法：在大模型工程中融合两者

在构建复杂的 Agent（智能体）时，我们通常会**用 Prompt Chain 搭建骨架，用 CoT 填充血肉**。

例如，开发一个“自动写代码并测试”的 Chain：

- **Node 1 (Chain)**：根据需求生成设计文档。
- **Node 2 (Chain)**：根据设计文档写代码。**（在这一步的 Prompt 内部，使用 CoT 引导模型先分析边界条件、时间复杂度，然后再写代码）**。
- **Node 3 (Chain)**：编写测试用例。

## 四、 资深工程师必备：CoT 的高级演进变体

随着技术的发展，标准的 CoT 已经演化出了几个更强大的高级变体，这是资深 Prompt 工程师在面对极其刁钻的任务时必须掌握的武器：

### 1. ToT (Tree of Thoughts，思维树)

标准的 CoT 是线性的（Step 1 -> Step 2 -> Step 3）。但人类思考会分叉，会走弯路。
ToT 允许模型在每个推理节点上**生成多个可选的思维分支**，并引入一个“评估器”（可以是模型自身）来给这些分支打分，通过剪枝、回溯（BFS/DFS 算法）来寻找最优解。适合解数独、写复杂算法。

### 2. GoT (Graph of Thoughts，思维图)

比树状更进一步，允许思维分支之间进行**合并（Merge）和循环（Loop）**。将不同的思考线索汇聚成一个最终方案。

### 3. Self-Consistency（自一致性采样）

这是提升 CoT 稳定性的工程神技。

- **做法**：设置模型的 Temperature > 0（增加随机性），让含有 CoT 的 Prompt 运行 $N$ 次（比如 5 次或 11 次）。
- **原理**：模型会生成 5 条不同的推理路径和 5 个答案。最后通过**投票（Majority Vote）**，选择出现频次最高的那个答案。
- **优势**：极大地降低了单次推理由于“手滑（Token 预测偏差）”导致的错误。

## 五、 CoT 的工程落地避坑指南

1. **不要在简单任务上使用 CoT**：
   如果任务是“将这段文字翻译成法语”或“提取以下文本中的人名”，使用 CoT 除了增加 Token 成本、提高延迟（Latency）之外，没有任何好处，甚至可能因为说多错多而降低准确率。
2. **模型的体量决定了 CoT 的上限**：
   CoT 是一种**涌现能力（Emergent Ability）**。通常在参数量较大的模型（如 Gemini 1.5 Pro, GPT-4 级别）上表现极佳。如果在极小的端侧模型上强制使用 CoT，它可能会陷入逻辑循环（鬼打墙）或生成毫无意义的废话。
3. **注意当前大模型（如 2026 年的主流模型）的内置原生思维**：
   现在许多前沿模型已经内置了**Extended Thinking（扩展思考/推理模型）**机制。在使用这类模型时，你甚至不需要显式书写 CoT 提示词，模型在底层已经自动进行了深度多步推理。此时，你的 Prompt 应当更专注于**定义清晰的边界条件和输出格式**。
