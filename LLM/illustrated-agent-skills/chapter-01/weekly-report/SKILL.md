# 写周报

把用户列出的本周工作整理成一份语气亲和、像同事之间汇报的 Markdown 周报，保存到当前工作目录。

## 用户会怎么告诉你

用户会列出本周做了哪几件事，通常会顺手在每条后面用括号标耗时，例如：

- 增加元素控制点，完成授权开发 (2d)
- 前端国际化走查问题修复 + 国际化 Skill 优化 (1.5d)
- 白牌定制需求开发 (1d)

耗时格式可能是 `(2d)`、`(0.5d)`、`(3h)` 等。**按用户原样保留，不要做单位换算或四舍五入**——人家自己记的工时比你估的准。

如果某条任务用户没带耗时，**不要自己估**。先问一句"X 这条大概花了多久？"再继续。原因：周报是给老板/同事看的，凭空捏的工时会让汇报失真，用户也可能因此尴尬。

## 时间范围（本周一 ~ 本周五）

自动按用户当前日期算"本周一到本周五"。当前日期可以从环境上下文里的 `Today's date` 取得，不要去问用户。

判断逻辑：
- 取今天的星期几。Monday=1, ..., Friday=5, Saturday=6, Sunday=7（ISO 编号）。
- 本周一 = 今天 - (weekday - 1) 天
- 本周五 = 本周一 + 4 天

例：今天是 2026-06-16（周二），本周一是 2026-06-15，本周五是 2026-06-19。

**周末的边界**：如果碰到周六或周日才写周报，仍然算"本周一~本周五"（即刚过去的那一周），而不是下一周——因为这时候用户多半是在补交本周工作汇报。

跨平台日期计算可以用 Python 一行解决，避免 PowerShell / bash 的差异：

```bash
python -c "from datetime import date,timedelta; t=date.today(); m=t-timedelta(days=t.weekday()); print(f'{m}~{m+timedelta(days=4)}')"
```

## 输出格式

严格按这个模板，**不要自作主张追加"下周计划"、"心得体会"、"风险提示"等小节**——用户明确说只生成这两块，加了反而显得啰嗦。

```markdown
# 周报（YYYY-MM-DD ~ YYYY-MM-DD）

Hi 大家好~ 同步一下本周的工作进展：

- 任务描述 (耗时)
- 任务描述 (耗时)
- 任务描述 (耗时)

有任何问题欢迎随时找我聊，祝大家周末愉快～
```

写作要点：

- 开头和结尾各一句口语化客套，让整体像在群里发汇报，而不是冷冰冰的清单。可以稍作变体，不必每周都一字不差。
- 任务条目**保留用户原话**，不要润色改写、不要合并拆分、不要补充"详情"。用户写啥就是啥，他自己拿捏措辞的分寸。
- 耗时按用户给的格式原样追加到行尾，例如 `(2d)`、`(0.5d)`、`(3h)`。
- 不要加表情符号堆叠或夸张的感叹号，保持职业感。一两处轻量的 `~` 已经足够亲和。

## 保存文件

文件保存到**当前工作目录**，文件名格式：`weekly-report-YYYY-MM-DD.md`，其中 `YYYY-MM-DD` 是本周一的日期。

例：本周是 2026-06-15 ~ 2026-06-19，文件名就是 `weekly-report-2026-06-15.md`。

如果文件已存在，直接覆盖即可——一般是用户当天补内容或调整任务列表。

## 完整示例

**用户说：**

> 帮我写周报。本周做了：
> - 授权控制点开发 (2d)
> - 前端国际化走查问题修复 + 国际化 Skill 优化 (1.5d)
> - 白牌定制需求开发 (1.5d)

**假设今天是 2026-06-16（周二），生成的文件 `weekly-report-2026-06-15.md`：**

```markdown
# 周报（2026-06-15 ~ 2026-06-19）

Hi 大家好~ 同步一下本周的工作进展：

- 授权控制点开发 (2d)
- 前端国际化走查问题修复 + 国际化 Skill 优化 (1.5d)
- 白牌定制需求开发 (1.5d)

有任何问题欢迎随时找我聊，祝大家周末愉快～
```

**生成后回复用户**：用一两句话告诉用户文件保存在哪、内容大概是什么，方便他点开复核。不要把整份周报正文再贴一遍——文件里已经有了，重复显得啰嗦。
