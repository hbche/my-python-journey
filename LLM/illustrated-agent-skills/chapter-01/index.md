# 第一章 先用用：跑通你的第一个 Skill

通过技能实现对LLM的掌控能力比提示词更强。

## 1.1 环境准备

使用 Claude Code 或 Coze(扣子) 等支持 Skill 的智能体

## 1.2 跑通第一个技能

### 1.2.1 不用技能

在智能体的对话框里输入下面这段话：

```md
帮我写一份本周周报。本周做了三件事：

1. 授权控制点开发
2. 前端国际化走查问题修复 + 国际化 Skill 优化
3. 白牌定制需求开发

要求：表格格式，三列（任务、进度、备注），语气亲和，保存成 Markdown 文件。
```

我们每周都得手动重复地给智能体发送上面这些 Prompt，反复向智能体发送重复的指令，耗时、好礼且输出可能不稳定。

### 1.2.2 创建技能

两种典型的方式

#### 1. 需求清晰，直接创建

智能体自带 skill-creator 技能，可以让智能体依据该技能根据我们的指令创建我们的技能。

使用大白话描述我们的需求：

``` md
我想创建一个写周报的技能。每次我告诉它本周做了什么，它就帮我生成一份周报。
格式要求：
本周工作日时间范围，例如，2026-06-08~2026-06-12
本周工作内容，将我告诉你的内容以无序列表形式展示，并在每条数据项末尾追加耗时，例如：增加元素控制点，完成授权开发 (2d)
语气亲和，像同事之间汇报工作。最后保存成Markdown文件。
```

#### 2. 先对话调试，再固化成技能

如果我们不清楚周报具体需要做成什么样，没关系。我们先跟智能体以聊天方式做一份出来，边做边调，调到达到我们语气效果之后再固化成技能。


智能体通过回顾我们刚才的全部对话，我们中间提出的每一个调整到最终满意结果，智能体会自动提炼规则，生成技能文件。这比我们从零开始描述需求更准确，因为智能体目睹了完整的调整过程。

> 智能体只能看到当前对话的历史。如果我们退出对话后，之前聊天的内容它会遗忘。所以“调试满意 -> 固化成技能” 这两步必须在同一个对话里完成，只有这样，智能体才能提炼到刚才对话过程中产生的全部要求。

#### 3. 两种方式，两种场景

对于执行某项任务，方式一适合你已经有明确标准的情况，即需求清晰，一句话直接创建技能；方式二适合你还在摸索的情况，先做出满意的结果，再让智能体把最佳实践固化成技能。通过这两种方式创建出来的技能文件格式完全一样，效果也一样。

#### 4. 差异在哪里

没有技能的话，我们每次都需要将相同的指令重复的发送给智能体，并且无法保证生成效果是一致的。

#### 5. 尝试一下

```` txt
根据本周工作内容
```
入侵 2.18.0 版本 可管理控制点需求开发 3d
前端国际化走查/skill 优化 1d
无相 1.8.0 版本 需求开发 0.5d
```
帮我生成一份周报
````

输出不满意的话，直接让正难题修改技能。比如

``` md
1. 标题直接改为时间范围，不需要“周报”，也不需要括号
2. 标题需要设为2级
```

### 1.2.3 技能没有生效

1. 确认技能是否已经安装
2. 确认智能启已重启
3. 检查技能description，确认我们发送的指令是否命中技能

### 1.3.1 一个文件夹，一个文件

### 1.3.2 SKILL.md 的三个部分

- name: 技能的名称，也是文件夹的名称，也是技能的ID，智能体通过这个ID标识不同的技能
- description: 技能的简介，一两句话说清楚“技能是干什么的，以及什么时候使用”
- 正文: 具体的操作步骤和规则，其中可能包含执行流程、输出要求和检查清单

### 1.3.3 技能存放在哪里

1. 云端智能体
2. 本地智能体
