# 第 2 章 理解技能运行的核心逻辑

本章目标：

1. 认识智能体的6大要素
2. 理解技能相比提示词强在哪里
3. 理解上下文和按需加载机制
4. 学会写好技能的 description

## 2.1 智能体的六大要素

1. Prompt
2. 上下文
3. LLM
4. 工具
5. 技能
6. MCP

### 2.1.1 提示词：对厨师下的指令

向大模型下发个性化要求，控制大模型概率游走空间的范围。

提示词最大的特点是临时性，不具备夸对话自动生成的能力。

技能跨平台通用。

### 2.1.2 大模型：厨师

本质是一个概率空间，依托概率计算进行下一个 Token 预测，输出概率最高的。

### 2.1.3 上下文：厨房台面

上下文是大模型的工作记忆，与智能体的对话、当前加载的技能名片、调用工具返回的结果等，都临时存放在这块空间。

这块记忆空间是有限的，一旦空间满了，先前存入的内容会被挤掉，大模型就开始“忘事”。

### 2.1.4 工具：厨房用具

- 内置工具：文件读写、执行代码、搜索网页
- 扩展工具：接入外部服务，如操作浏览器、链接数据库、对接飞书

### 2.1.5 技能：菜谱

技能就像写给厨师看的菜谱，是写个大模型使用的操作手册。

有组织的文件集合。

智能体可以拆分成两部分：核心的“大模型”和提供支持的“智能体运行框架”（agent harness）。大模型像大脑，负责思考、推理和下发指令；智能体负责加载技能、管理上下文、调度工具的底层程序，共同组成了智能体框架。

### 2.1.6 MCP：通用插座

智能体对接外部工具，比如数据库、邮件系统、日历、飞书等工具，这些工具如果都要单独适配的话，则很难使用。MCP就是为上述工具的接入提供一个统一接口。

MCP是一套统一的对接标准，让不同的外部工具，都使用同一种方式接入智能体。提供工具的厂商只要实现MCP协议，就可以接入所有支持MCP的智能体。

工具让智能体动手干活，技能教智能体具体怎么做，MCP让各种外部工具无缝接入。

## 2.2 技能比提示词强在哪里

技能比提示词强，并不是说提示词“落后”了。提示词永远是人与大模型交互的基础。这里被拿来与技能作对比的，是那些试图充当“操作手册”的提示词（以及提示词模板），也就是那些为了限制AI边界，而把几千字的业务流程等诸如此类的内容，一股脑儿硬塞进上下文时所用的那种长篇大论的提示词。

### 2.2.1 为什么需要技能

提示词有两个硬伤：

1. 极其挤占“工作记忆”：几千字的提示词一旦塞进上下文，就会迅速耗尽上下文窗口的有限容量，导致大模型注意力分散，很容易“看漏行”忘掉后半段的指令。
2. 无法稳定绑定复杂的工具流：如果你的报告不仅需要排版，还需要智能体先去数据库拉取最新价格，再运行一段Python代码画折线图，单纯的文本提示词是很难让它连续且稳定地完成这些复杂的工具调用的。

从更深的层面看，智能体当前的智能已经足够了，但不了解我们的业务规范。我们平时摸索出来的流程、踩坑总结这类知识都属于“流程性知识”，这个是大模型不知道的。建立技能就是把我们大脑里的隐性经验整体封装成AI的SOP（standard operating produce，标准作业程序）。

### 2.2.2 四大进化

1. 打破全量塞入的容量限制，实现按需加载
2. 突破上下文边界，让文件系统称为持久化工作台
3. 摆脱单打独斗，构建协同工作流
4. 告别碎片化维护，积累经验复利

|          | 提示词                             | 技能                                             |
| -------- | ---------------------------------- | ------------------------------------------------ |
| 加载方式 | 全量塞入上下文                     | 按需加载                                         |
| 工作空间 | 只能在上下文中，对话结束即消失     | 文件系统可作为附加工作台，中间产物可存可取可重跑 |
| 协作能力 | 单打独斗，模板之间互不相同         | 可串成工作流，可带脚本处理确定性任务             |
| 迭代积累 | 模板散落各处，改一份，别处还是旧版 | 迭代同一份文件，后续任务自动用最新版             |

## 2.3 上下文按需加载

### 2.3.1 对话越长，可用空间越少

与AI聊着聊着，AI就变“笨”了，这背后有两个原因：

1. 工作记忆空间被历史对话占满，新信息实在挤不进去。
2. 即使空间还没满，其中堆积的信息越多，大模型的“注意力”就越容易被分散，导致它顾此失彼，开始遗漏关键指令。

### 2.3.2 按需加载：用到什么才拿什么

官方叫法是渐进式披露：系统不会把所有技能文件一股脑儿塞进上下文，而是将其拆分成三层。智能体在接到我们的指令后，会按需逐层追加。我们以写周报为例讲解：

#### 第一层：技能名片（YAML前置信息）

智能体刚启动时，塞进上下文的只是各个技能的名片，即YAML前置信息，它浓缩了两类信息。

- name: weekly-report
- description: 生成本周工作周报的 Markdown 文件。当用户说"帮我写周报"、"本周做了 X、Y、Z"、"周总结"、"汇报本周工作"、"工作汇报"，或列出几件本周完成的事情并希望整理成文档时，使用此 skill。即使用户没有明确说"周报"两个字，只要意图是把若干本周做完的工作整理成同事之间汇报的形式，也应该触发。

##### 触发动作与开销

当我们说“帮我写周报”时，智能体的第一个动作是拿我们的需求跟所有技能的description做比对匹配，判断该用哪个技能——鉴于description的作用类似于开关，我们又把它称为“触发开关”​。因为名片早就加载好了，所以这一步不消耗额外的词元量。又因为名片的体量比较小，就算加载一二十个也不会轻易“撑爆”上下文窗口。

#### 第二层：技能核心说明书（SKILL.md正文）

确定该用哪个技能之后，在第一层信息的基础上，智能体开始读取技能相应的SKILL.md的正文。

##### 触发动作与开销

一旦在第一层锁定了具体的技能，智能体会在后台调用一次文件读取工具，把完整的SKILL.md加载到上下文。

请注意，这意味着触发技能是有代价的，它消耗了一次工具调用。更重要的是，这就相当于把SKILL.md的全部内容加载为上下文——大模型需要将其作为“历史聊天记录”来理解任务。由于大模型会记住所有的历史对话，在这次对话结束前，它会一直挤占宝贵的上下文空间。

像周报这样的技能结构简单，有这两层就够了。但随着我们做的技能越来越复杂，第三层就派上用场了。

#### 第三层：技能配套资源包（关联文件）

除了最核心的SKILL.md，有些技能还附带了一个配套资源包，Anthropic官方将这些文件统称为关联文件(linked files)。这里面装的是遇到复杂任务时才需要的“扩展弹药”​。请注意，关联文件不局限于 .md格式，它可以包含脚本、参考文档和静态资源。

比如，我们做了一个处理PDF文件的技能，它能从PDF里提取文字和表格、填写表单、合并多个文档。如果把所有的操作说明全塞进一个SKILL.md，这个文件会变得极其臃肿。更好的做法是，让SKILL.md只负责讲清楚核心的处理流程，而把繁杂的细节拆分出去。

##### 参考文档

- 位置：`references/` 目录
- 内容：把“填写PDF表单”这种极其琐碎的操作规范，拆分成独立的forms.md文件。只有当遇到填表任务时，forms.md才会被大模型调出来查阅。一些其他高级功能和完整的API，则可以放到reference.md文件中。

##### 脚本

- 位置：`scripts/` 目录
- 内容：把真正用来“提取表格”和“合并文档”的繁重计算，写成独立的pdf_tools.py脚本，交由智能体框架在后台直接运行，不用大模型操心。

##### 静态资源

- 位置：`assets/` 目录
- 内容：在这里附带一份standard_template.pdf作为表单的标准样板，或者放一张logo.png，在合并文档时让脚本自动打上水印。

通过这种拆分，SKILL.md保持了极简，而关联文件则组成了一个功能完备的“外部工具箱”​。

> SKILL.md 如何“牵出”关联文件？
>
> 拆分之后，智能体怎么知道该在什么时候去读哪个关联文件？答案很简单：在SKILL.md的正文里，用大白话告诉它“如果遇到什么情况，就去读哪个文件”​（条件+ 相对路径）​。

前面提到相对路径的问题，我们再强调一遍：引用关联文件时，请使用从技能根目录出发的相对路径（如references/reference.md、scripts/extract.py）​，并且只引用一层。不要出现A引用B，B又引用C的嵌套链——层级越深，智能体越容易迷路，阅读开销也越不可控。一句话，引用路径要浅，不要嵌套。

##### 触发动作与开销

- 对于参考文档（按需加载）：智能体只有在执行到对应步骤时，才会触发读取动作。不走到那一步，绝不加载，真正做到“零前置消耗”​。
- 对于脚本（沙盒运行）：繁重的计算过程完全在上下文之外的“独立沙盒”中运行，绝不挤占大模型的“工作记忆”​。

| 层次   | 加载内容                             | 触发动作                 | 加载时机                                   |
| ------ | ------------------------------------ | ------------------------ | ------------------------------------------ |
| 第一层 | name、description及其他选填信息      | 通过比对选出最匹配的技能 | 启动即加载，每个技能只占几句话长度的词元量 |
| 第二层 | SKILL.md正文                         | 调用工具读取             | 判断需要改技能时加载                       |
| 第三层 | 关联文件（脚本、参考文档、静态资源） | 按需读取或沙盒运行       | 执行过程中必要时才加载                     |

```md
your-skill-name/ # 技能文件夹（技能包）
├── SKILL.md # 技能主文件（必填）
├── scripts/ # 脚本（可选）
│ ├── process_data.py # 示例文件
│ └── validate.sh # 示例文件
├── references/ # 参考文档（可选）
│ ├── api-guide.md # 示例文件
│ └── examples/ # 示例文件夹
└── assets/ # 静态资源（可选）
└── report-template.md # 示例文件
```

### 2.3.3 管理上下文的三个技巧

#### 技巧1：按需启用，严格控制常驻技能的数量。

- 官方建议安装的技能数量的上限为20~50个。
- 分层管理技能，分云端、本地全局技能、本地项目技能

#### 技巧2：主文件（SKILL.md）必须克制

既然技能一旦触发就收不回去，那就千万别把它写成几万字的百科全书。

很多新手喜欢把厚重的背景知识甚至FAQ(Frequently Asked Questions)全塞进SKILL.md。结果技能一触发，大量挤占上下文。这不仅会导致剩余的计算空间变少，还会严重稀释大模型的注意力，最终导致对话质量直线下降。

官方明确建议：将SKILL.md的正文长度控制在5000个词元以内（换算成中文，大约不超过3800字）​，行数不超过500行。你应当只把最核心的操作流程和规范写在主文件里；而对于完整的API规范等重型内容，务必拆分到关联文件中，利用前面讲过的“条件 + 相对路径”的方式（见“SKILL.md如何‘牵出’关联文件？​”​）​，让智能体按需读取。

#### 技巧3：适时开启新对话以“清空工作记忆”

SKILL.md主文件一旦被读取，就会一直留在那一轮对话的上下文中，挤占工作记忆空间，直到对话结束。如果在一次长对话过程中，你连续触发了好几个复杂的技能，这种“资源占用”的累积效应极其可怕。

遇到智能体回复变卡、输出质量下降，或者它开始“忘掉”最初的指令时，最简单、最彻底的解法就是：直接开启一个新对话。这相当于一键清空工作记忆，让智能体轻装上阵，重新开始。

## 2.4 description：技能被选中的关键机制

### 2.4.1 智能体是如何相中技能的

技能的元数据在智能体启动时就已经预加载到本轮对话的上下文中了，所以 description 是写给智能体看的。

#### 触发开关如何发挥作用

智能体将我们输入的指令与预加载的技能的元数据中的description进行比对推理，不是精准匹配，是大模型使用自己的推理能力去判断，然后决定提取命中技能的SKILL.md正文。

### 2.4.2 为什么有时候会选错

既然description是核心决策依据，那它就不只关乎功能，更关乎效率。激活技能本身会消耗一次工具调用，description写不好会带来两种代价。

- 误触发：智能体被模糊的描述误导，加载了一个不相关的技能，白白浪费了一步操作，还无端占用了宝贵的上下文。
- 漏触发：明明装了技能，智能体却没认出来，自己“即兴发挥”去干活了，但即兴发挥的结果是不稳定的。

### 2.4.3 如何写出好的 description

#### 1.极简公式

description = 功能定义 + 触发场景/触发词

- 功能定义：这个技能是用来干什么的
- 触发场景/触发词：什么时候使用功能这个技能

#### 2.两个陷阱

- 人称混乱：建议去掉人称代词，以陈述句直接描述技能的功能
- 描述太长：对于通用场景技能的description，推荐50~200个字符；对于大数量技能集的description，按规模控制在130/150个字符内。写description的核心原则为“精简、精准，不冗余”​

#### 3.高阶技巧：使用反向触发词避免技能“抢活儿”

例如润色技能和翻译技能可能抢着触发，这时候就需要使用反向触发词，明确划定界限。

##### 基础写法

```md
description: 将文章翻译为目标语言。当用户需要翻译时使用
```

##### 加反向触发词

```md
description: 将文章翻译为目标语言。当用户需要翻译文章、网页内容时使用。不用于润色已有中文内容，不用于改写或摘要
```

反向触发词不用写太多，一句话点明最容易混淆的场景就够了。装三五个技能通常用不上反向触发词，装20个技能以后你会感谢自己加了。

#### 4.速查清单

| 检查项                        | 合格标准                                      |
| ----------------------------- | --------------------------------------------- |
| 极简公式                      | 功能定义 + 触发场景/触发词，缺一不可          |
| 触发词                        | 覆盖用户可能使用的关键词（同义词、中英文）    |
| 反向触发词                    | 如果安装了多个功能相近的技能，加上“不用于...” |
| 人称正确                      | 用无人称陈述句，不用“我”“你”                  |
| 长度合理                      | 100字左右，不超过200字                        |
| 触发条件只写在 description 中 | 没有把与触发相关的信息写到 SKILL.md 的正文里  |
