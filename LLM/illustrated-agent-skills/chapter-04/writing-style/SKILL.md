# 写作风格：hanbin.che 的个人写作风格

把你的文字改写成 hanbin.che 的风格——像一位技术导师兼同事在跟你聊天，用大白话把技术讲清楚。

## 四大维度速查

### 用词偏好

核心原则：**能说人话就说人话，能用动词就不用名词，能用短词就不用长词。**

- 口语动词优先："干活"、"搞定"、"聊聊"、"试试"、"动手"、"塞进"、"挤占"、"串起"、"摊开"、"拧开"
- 技术术语后必跟大白话翻译：术语出现 → 破折号或"也就是" → 一句人话解释
- 用一套饮食/厨房比喻贯穿全文：菜谱=指南、厨师=模型、厨房台面=上下文、菜市场=外部工具、买鲈鱼=调外援
- 方言和俚语适度点缀："断片儿"、"摸不着北"、"即兴发挥"、"无脑点"
- 双字组合收尾造节奏："耗时、好礼"、"精简、精准，不冗余"、"该严的严，该松的松"
- 绝对禁止的词：不能出现"赋能"、"抓手"、"闭环"、"颗粒度"、"底层逻辑"这类互联网黑话

### 句式习惯

核心原则：**一句只说一件事，能用短句绝不用长句。**

- 单句控制在 5-25 字，超过 30 字的句子拆成两到三句
- 并列结构用分号或顿号制造节奏，不用"以及"、"和"连接长串
- 高频句式"不是...而是..."用于纠偏和定位：先否定常见误解，再给出准确定义
- 设问开头驱动段落：每节开头用"为什么..."、"怎么..."、"问题在哪..."自问自答
- 排比收束做总结："提示词做 X，脚本做 Y，技能做 Z"——三条一组，句式一致
- 祈使句给指令："别..."、"千万..."、"先...再..."——直接、简短，不加"请注意"这类客套前缀

### 结构特点

核心原则：**先给地图再带路，每层都有导航。**

- 固定三段式：目标 → 正文 → 小结。目标列出本章要掌握什么，小结用编号回顾要点
- 每节内四步走：概念定义 → 生活比喻 → 技术原理解释 → 实操建议
- 表格做速查：对比信息、检查清单全部用表格承载，三到四列表格为佳
- 引用块标警告：重要的"注意"、"避坑"、"限制条件"用 > blockquote 单独拎出
- 示例紧跟概念：讲完一个规则立刻跟代码块或示例文本，不吊胃口
- 编号层级不超过四级（# → ## → ### → ####），再深就拆新节

### 语气风格

核心原则：**导师 + 同事的双重身份，权威但不居高临下。**

- 用"你"直接对读者说话，用"咱们"制造同盟感
- 先正面鼓励再泼冷水："先跑起来，再慢慢调" → 紧接着说清楚前提和限制
- 过来人口吻："很多新手喜欢...结果..."——用踩坑经验提醒，不是教科书警告
- 实用主义判断标准：不问"好不好"，只问"值不值得"、"省不省时间"、"能不能复用"
- 结尾轻量收束：章节末尾用一句口语化总结，不加煽情拔高，不加感叹号堆叠
- 一两处轻量的 `~` 已经足够亲和，绝不用表情符号轰炸

## 写作自检清单

写完一段后，用这张表快速过一遍：

| 检查项          | 合格标准                                          |
| --------------- | ------------------------------------------------- |
| 句子长度        | 没有超过 30 字的单句                               |
| 术语解释        | 每个陌生术语后面都跟着大白话解释                   |
| 比喻一致性      | 全文使用同一套比喻体系，不混用                     |
| 互联网黑话      | 零出现                                            |
| 结构完整性      | 有目标、有正文、有小结                             |
| 表格使用        | 对比类信息用表格呈现，三到四列                     |
| 警告可见性      | 重要提示用 > blockquote 单独标出                   |
| 示例密度        | 每条规则后面至少跟一个示例                         |
| 语气一致性      | "你"+"咱们"贯穿，不突然切换到"笔者"、"我们"       |
| 结尾克制度      | 不煽情、不拔高、不用感叹号轰炸                     |

## 参考示例

以下是符合本风格的典型段落：

```markdown
模型擅长理解和生成，但模型是基于概率预测来工作的，做规则固定的事偶尔会出错。

规则清晰、要求明确的活儿很适合交给脚本。

|          | 交给模型                   | 交给脚本                   |
| -------- | -------------------------- | -------------------------- |
| 哪些活儿 | 需要理解、创意和分析       | 规则明确、步骤固定         |
| 典型任务 | 分析趋势、撰写结论、做决策 | 数据清洗、格式转换、文本替换 |
| 开销     | 挤占上下文，消耗 Token     | 在上下文外执行，几乎不消耗 Token |
| 结果特点 | 每次略有不同               | 每次完全一样               |
```

**风格要素拆解：**
- 用"活儿"替代"任务"，用"偶尔会出错"替代"存在不确定性"
- 短句开路讲概念，表格做对比总结
- 对比结构先肯定再否定："擅长 X，但 Y"
- 表格四列精简，每格不超过 15 字
