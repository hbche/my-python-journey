# 业界国际化技术方案：从"能用"到"好用"

## 目标

1. 搞懂国际化到底在解决什么问题
2. 认识业界主流的几套方案，知道什么时候选哪套
3. 理清前端和后端各自的活儿怎么分
4. 避开最常见的几个坑

## 1 国际化到底在忙活什么

国际化（i18n）听上去挺唬人，说白了就一句话：**同一套代码，撑起不同语言和地区的用户**。

> 把国际化叫"i18n"是因为 internationalization 这个词，首字母 i、尾字母 n，中间夹了 18 个字母。类似的还有本地化（l10n），指针对某个具体地区的适配。这两件事常被混着说，但干活的时候得分清：国际化是"让代码能撑得住多语言"，本地化是"给中国用户换上中文、给美国用户换上英文"。

它不只是"把按钮上的文字翻译一下"。拆开来看，最少要搞定四块：

| 要搞定的 | 什么意思 | 举个例子 |
| -------- | -------- | -------- |
| 文字翻译 | 界面上的所有文字能切换语言 | 登录 → Login |
| 格式适配 | 日期、时间、数字、货币按当地习惯展示 | 2026/06/16 → 06/16/2026 |
| 布局适配 | 文字方向、长度变化后界面不崩 | 英文"Save"→ 德语"Speichern"，按钮得跟着变宽 |
| 内容地区化 | 不同地区看到不同的功能或数据 | 国内用微信登录，海外用 Google 登录 |

## 2 三套主流方案，各有各的适用场景

业界搞国际化，归根到底就三套路子。选方案之前，先看清每套的脾气。

### 2.1 资源文件式：最老派，也最稳

这是最经典的做法——把每句话的翻译存到文件里，代码里用 key 去取。

就好比饭店后厨的备料单：中餐版一页、西餐版一页，厨师炒菜时看对应的那一页。

```json
// zh-CN.json
{ "greeting": "你好", "submit": "提交" }

// en-US.json
{ "greeting": "Hello", "submit": "Submit" }
```

```javascript
// 使用时用 key 去取
t('greeting') // 中文环境返回"你好"，英文环境返回"Hello"
```

主流选手：

| 选手 | 擅长什么 | 适用场景 |
| ---- | -------- | -------- |
| react-i18next / vue-i18n | 前端框架深度绑定，模板里直接用 | React / Vue 项目 |
| i18next | 通用 JS 方案，前后端都能跑 | 中大型项目，需要插件扩展 |
| FormatJS (react-intl) | 日期数字格式化能力极强 | 对格式要求精细的场景 |
| gettext | C/PHP/Python 老牌方案 | 后端、桌面软件 |

> 资源文件式的核心武器是 ICU MessageFormat。它是 Unicode 联盟定的一套消息格式标准，能搞定单复数、性别、选择这些复杂语法。比如 `"我有 {count, plural, =0 {零个苹果} =1 {一个苹果} other {# 个苹果}}"` ——一段文本搞定所有复数形式，不用写 if/else。

**怎么选？一句话**：项目不大不小、翻译量可控、没有专门的翻译团队——用这套就够了。

### 2.2 翻译平台式：把翻译这件事外包出去

当翻译量滚到几千条甚至上万条，光靠开发者手改 JSON 文件就是在给自己找麻烦。

翻译平台做的事：把你的 key 和源语言自动同步到云端 → 翻译人员在线翻译 → 翻译结果自动拉回代码仓库。开发者不用再手动排版、催进度、对条目。

就好比餐厅做大了，菜单翻译不再让厨师自己翻词典，而是交给专门的翻译公司，厨师只负责炒菜。

主流选手：

| 选手 | 特点 | 适合谁 |
| ---- | ---- | ------ |
| Lokalise | 全功能，支持 CI/CD 自动同步 | 中型以上团队 |
| Crowdin | 开源项目友好，GitHub 直接集 | 开源、小团队 |
| Phrase | 翻译记忆库 + 质量检查做得好 | 翻译专业度要求高 |
| Tolgee | 开源、自部署可选 | 数据不能出海的场景 |

这些平台都有一个关键能力——翻译记忆库。同一句话翻过一次，下次遇到自动复用。翻译团队的效率能翻好几倍，省下来的钱够买好几年的平台年费。

> 选翻译平台有一个硬约束你先想清楚：**翻译数据存在哪里？** 如果是 SaaS，数据在平台方的服务器上。涉及内部业务系统的翻译、合同条款、敏感字段，务必选支持自部署的方案，或者在流水线里把翻译文件拉到本地再处理。

### 2.3 编译时方案：把翻译在构建环节就定死

前面的方案都是运行时做翻译——用户打开页面，代码才去查当前语言该显示什么。

编译时方案换个路子：**构建时就根据语言生成多个版本，用户访问时直接拿对应版本**。

就好比出版社同时印了中文版和英文版两本书，读者买到的就是自己语言的版本，不需要一边读一边查词典。

```bash
# 构建时一口气打出多个语言版本
npm run build -- --locale zh-CN  # 输出 dist/zh-CN/
npm run build -- --locale en-US  # 输出 dist/en-US/
```

| 对比维度 | 运行时翻译 | 编译时翻译 |
| -------- | ---------- | ---------- |
| 首屏速度 | 多一次翻译函数调用 | 直接用静态文本，更快 |
| 切换语言 | 瞬间切换 | 需要重新加载页面 |
| 包体积 | 所有语言打包在一起 | 每种语言一个独立包 |
| 部署复杂度 | 一次部署 | 每种语言部署一份 |

> 编译时方案有个硬伤：**语言一多，构建时间线性增长**。10 种语言就构建 10 次，每次构建都要走完整的打包流程。如果你的项目支持超过 5 种语言，建议在 CI 里并行跑构建，把每种语言拆成独立任务。

## 3 前端和后端，活儿怎么分

很多人上来就把所有翻译全扔给前端，后端只返回中文错误码。这么干有一个坑：**后端返回的错误信息、邮件模板、短信验证码、PDF 报表，前端根本管不着**。

### 3.1 前端干的活儿

前端负责所有"用户直接看见"的文字：

- 页面上的按钮、标签、占位符
- 表单校验的提示文字
- 日期、数字、货币的格式化
- 图片里的文字（不同语言切不同图片）

```javascript
// 前端典型的 i18n 用法
<button>{t('submit')}</button> // 中文环境显示"提交"
<span>{formatDate(date, { locale })}</span> // 按当地格式显示日期
```

### 3.2 后端干的活儿

后端负责所有"系统生成后推到前端的"文字：

- API 返回的错误提示（前端收到的不应该是前端自己猜的）
- 邮件、短信、站内通知
- 导出 PDF、Excel 里的表头和内容
- 枚举值的展示名称

```python
# 后端典型的 i18n 用法
raise ValidationError(_("用户名已存在"))  # 根据请求的语言返回对应翻译
```

### 3.3 一句话分工

| 负责方 | 管什么 |
| ------ | ------ |
| 前端   | 用户界面上直接看到的文字和格式 |
| 后端   | 系统生成的、推送到用户面前的消息和文件 |

> 最容易踩的坑：后端返回中文错误码（如 `USER_NOT_FOUND`），前端自己猜翻译。这有两个问题——第一，如果前端猜错了呢？第二，换了英文环境，后端日志里混着用户看不懂的错误码，排查起来极其痛苦。记住：**API 的返回信息应当对调用方友好，而不是对日志友好**。

## 4 动手前的五步思考

不是所有项目一上来就得把国际化做成全家桶。一上来就上全套平台 + 全语言覆盖，大概率是浪费。

### 4.1 先问自己三个问题

- **问题一：用户真的有多语言需求吗？** 如果你的产品只在国内卖，中文一套搞定，不需要为"万一以后出海"提前做国际化。提前过度设计跟提前优化一样，都是时间的黑洞。
- **问题二：翻译量有多大？** 100 条以内，JSON 文件手动管理绰绰有余。1000 条以上，老老实实上翻译平台。
- **问题三：谁来翻译？** 开发者自己翻的话，文件式就够了。有专门的翻译人员，平台式的协作能力才能发挥出来。

### 4.2 四种场景，四种搭配

| 场景 | 推荐方案 |
| ---- | -------- |
| 个人项目 / MVP | JSON 文件 + i18next，手动翻译 |
| 中小团队（< 20 人） | 文件式 + 一个轻量翻译平台（Tolgee / Crowdin 免费版） |
| 中大团队（20~100 人） | Lokalise / Phrase + CI/CD 自动同步 |
| 超大规模 / 微前端 | 翻译平台 + 运行时 + 编译时按需混用，子应用各自管理语言包 |

### 4.3 一个被忽视的时间炸弹

很多团队上来先只用中文写 key——比如 `"你好"` 作为 key。项目滚到几千条翻译后，key 和翻译文本混在一起，改一次翻译可能同时改了 key，其他引用的地方全炸。

```json
// 别这么干
{ "你好": "Hello" }

// 应该这么干
{ "greeting": "你好" }   // 中文包
{ "greeting": "Hello" }  // 英文包
```

> 用**语义化 key**，不要用展示文案做 key。key 一旦定下来就别再改，它是连接代码和翻译的桥梁。下次重构翻译文本时你会感谢当初的自己。

## 5 小结

1. 国际化不止翻译文字，还涉及格式、布局和内容地区化
2. 三套方案各有适用场景：文件式适合小项目，平台式适合有翻译团队的情况，编译时适合对性能极致追求的场景
3. 翻译量小用文件式，翻译量大用平台式，别一上来就过度设计
4. 前后端分工要清楚：界面归前端、消息归后端，别让后端日志污染前端体验
5. 用语义化 key，别用展示文案当 key——这是花五分钟省未来五小时的买卖
