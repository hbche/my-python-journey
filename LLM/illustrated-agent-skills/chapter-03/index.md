# 第 3 章 捋清楚：从任务到技能五步走

目标：

1. 学会拆任务，判断什么值得做成技能
2. 设计模型与脚本分工
3. 建立安全意识

不是所有任务都适合固化成技能，技能也不是越复杂越好。

动手前五步思考：

1. 一个任务该怎么拆？
2. 哪些环节值得固化成技能？
3. 哪些活儿交给模型？哪些活儿交给脚本？
4. 选什么工具干活？
5. 安全上要注意什么问题？

## 3.1 先把任务拆开看

在动手写技能之前，把任务里的每一个环节拆开看看。

### 3.1.1 任何任务无非拆成三类操作

做一桌菜之前，你得先分分类：哪些步骤照着菜谱来就行（比如“盐放 3 克”​）​，这属于**执行规则类操作**；哪些需要厨师看情况定（比如“看看今天的鱼新不新鲜，决定清蒸还是红烧”​）​，这属于**做判断类操作**；哪些食材厨房里没有，需要去外面买（比如“去菜市场买条鲈鱼”​）​，这属于**调外援类操作**。

- 执行规则：规矩定过一次，后面每次都一样。
- 做判断：没有固定答案，得看情况来。
- 调外援：智能体没有的工具，需要连外部。

### 3.1.2 拆完套路自然就出来了

### 3.1.3 拆得越清晰后面就越省力

拆任务可以避开以下三个坑：

1. 写出“万能技能”
2. 不知道哪里该严、哪里该松
3. 除了问题不知道该改哪里

#### 拆解你的第一个任务

1. 从日常工作职工选一个高频任务
2. 列出这个任务的每个环节，用下面的表格归类：
   |环节 | 执行规则 | 做判断| 请外援|
   | --- | --- | --- | --- |
   | | | | |
3. 看看哪类操作占比最大，它代表了我们这个技能的主要特征

## 3.2 要不要做成技能

在写技能之前，先思考这个任务值不值得做成技能。

做成技能有成本：创建要花时间、维护要花精力，每次触发还会消耗一次工具调用和占用上下文。

### 3.2.1 先问三个问题

- 问题一：这件事你会反复做吗？只做一次的事情，用提示词就够了。
- 问题二：你对结果的一致性有要求吗？对于头脑风暴、创意写作，这类任务不需要技能。
- 问题三：流程是否已经基本稳定？对于某件事，如果你还在摸索怎么做，就别急着写技能。过早固化意味着可能将错误的流程写进 SKILL.md。

做什么技能，一张表帮你找灵感。常见技能方向速查表：

| 所属套路                     | 具体方向                   | 一句话说明                                             | 典型应用                                                      |
| ---------------------------- | -------------------------- | ------------------------------------------------------ | ------------------------------------------------------------- |
| 标准化输出（主打“执行规则”） | 格式标准、风格约束         | 确保每次输出骨架一致、统一口吻和用词                   | 写周报、会议纪要、邮件                                        |
| 流程自动化（主打“做判断”）   | 多步流程、质量检查         | 串起多个环节，中间有检查点；拿着标准去挑刺并提出建议   | 文章配图、长文撰写、代码审查、译文校对                        |
| 工具增强（主打“调外援”）     | 外部数据分析、复杂工具使用 | 连接外部系统拿数据并出结论；教 AI 怎么用好某个复杂工具 | 数据库查询、生成转换漏斗报表、ffmpeg 视频处理、操作内网版飞书 |

不该做成技能的情况，别硬做成技能。具体来说有以下三种情况：

1. 始终生效的规则，放在全局配置里。“回复要简洁”​“代码缩进用 4 个空格”​，这类始终生效的规则应该放在智能体的全局配置里（Claude Code 用 CLAUDE.md，扣子没有相关的配置）​，不该做成技能。打个比方，​“所有菜都少放盐”是你家厨房的通用规矩，贴在墙上就行，不用写进每一份菜谱里。
2. 工具直接能搞定的事，用工具就好。“把图片转成 PNG 格式”​“把视频压缩到 10MB 以下”​，这类操作用 ImageMagick 或 ffmpeg 一行命令就能搞定，没必要包一层技能。这就像拧开水龙头就有水，这类操作不用写进菜谱里。技能的价值在于提供经验 ​，如果一个任务不需要你的经验就能完成，使用工具就够了。
3. 对于一次性任务，直接使用提示词。

### 3.2.2 再看四个限制

三个问题帮你判断“做不做”​。即使决定做了，也得心里有数：技能有四个限制，知道限制在哪里，能帮你做出更好的设计。

#### 1. 结果不完全确定。

问题：同样的输入不一定拿到同样的输出。你让技能生成数据分析报告，今天它可能把“销量下滑”放在第一条，明天可能换个顺序。这就像参考同一份菜谱，不同厨师炒出来的味道总会有细微差别。

解决办法：需要确定性的环节，拆出来交给脚本。数据清洗的规则、金额的精度、日期格式，这些写成代码，每次执行结果一模一样。就像厨师用上计时器和电子秤，跟自己估摸着来比，机器更准。

#### 2. 通用与可靠难两全

问题：规则写得越严格，执行越稳定，但适用范围就越窄。翻译技能里你规定“LLM、token、prompt 一律保留英文”​，技术文章的术语一致性就有了保障，但拿这个翻译技能去翻译一篇面向普通读者的科普文章，满篇英文术语反而可能让人读不懂。就像一份四川火锅底料的配方再精准，你也不能拿它去做日式寿喜锅。

解决办法：该严的严，该松的松，别追求一套规则通吃。有明确对照表的环节（术语处理）写死规则；因文章而异的环节（语气、句式）只给方向（比如“译文要自然流畅”​）​，让智能体自己判断。这就是 3.1 节拆解的价值——拆开了才知道哪里该严，哪里该松。

#### 3. 触发不一定准

问题：技能写得再好，智能体没在正确时机选中它，一切白搭。

解决办法：description 要反复测试

#### 4. 不同模型表现不同

问题：同一个技能在不同的模型上跑，效果可能差不少。就像同一份菜谱，交给大厨和新手，做出来的水平差别很大。

解决办法：复杂技能建议分别测。能力强的模型可能不需要那么多细节指引，能力弱的模型则可能需要更具体的步骤拆解。

### 3.2.3 最稳的起步方式

## 3.3 哪些活儿交给模型，哪些活儿交给脚本

需要做判断的交给模型，需要执行规则的交给脚本。

### 3.3.1 为什么不能全交给模型

模型擅长理解和生成，但模型是基于概率预测来工作的，做规则固定的事偶尔会出错。

规则清晰、要求明确的活儿很适合交给脚本。

模型与脚本分工

|          | 交给模型                   | 交给脚本                                                                           |
| -------- | -------------------------- | ---------------------------------------------------------------------------------- |
| 哪些活儿 | 需要理解、创意和分析       | 规则明确、步骤固定                                                                 |
| 典型任务 | 分析趋势、撰写结论、做决策 | 数据清洗、格式转换、文本替换                                                       |
| 开销     | 挤占上下文，消耗 Token     | 在上下文外执行，在大部分情况下，仅将处理状态或结果摘要返回上下文，几乎不消耗 Token |
| 结果特点 | 每次略有不同               | 每次完全一样                                                                       |

### 3.3.2 完整案例：出数据分析报告怎么分工

### 3.3.3 提示词、技能、脚本

- 提示词：语义驱动，适合一次性、简单、高度灵活的任务。
- 脚本：规则驱动，适合逻辑完全固定、追求 100%确定性操作。
- 技能：混合驱动，适合反复做、多步骤串联、既有固定规则又需要灵活判断的任务。

技能的核心价值在于填补了中间地带。

1. 比提示词更具约束力：通过结构化框架锁定目标，防止模型在多步骤中“断片儿”或产生幻觉。
2. 比脚本更具适应性：能够处理非结构化语义信息，这是死代码无法覆盖的领域。
3. 比传统系统更轻量：无须复杂部署即可实现可控、可追踪的流程编排

## 3.4 选择什么工具进行干活

确定哪些活儿交给模型，哪些活儿交给脚本，下一步是选择工具来执行。

### 3.4.1 先看手头有什么

智能体自带读写文件、代码执行、搜索网页的内置能力。很多任务光靠这些就够了。

如果内置工作不够，按这个顺序考虑如何安排扩展工具：

1. 有没有现成的命令行工具
2. 有没有现成的代码库
3. 需要连接在线服务吗？这时候需要考虑 MCP

### 3.4.2 别从零写，现成的库多的是

技能里的脚本用什么语言？Python 和 Node.js 是最佳搭配。

在技能里写清楚“这一步使用 x 库来处理”，智能体会自动帮你安装并编写脚本来调用。不过一定要注意库名必须拼写正确。

### 3.4.3 工具给能力，技能给经验

技能的核心并不是教智能体“如何使用工具”。

拿 ffmpeg 举例。智能体知道 ffmpeg 能处理视频，也能读 --help 了解参数。但它不知道你们团队的视频规范是什么：输出分辨率要 1080p、码率上限是 5Mbps、格式必须是 MP4、音频保留立体声……

技能的角色：“在我们这里，该怎么用好这个工具”的独家经验写下来。工具提供能力，技能提供经验，两者配合才能把活儿干好。

## 3.5 安全是头等大事

### 3.5.1 三道安全闸

1. 第一道闸：最小权限
2. 第二道闸：二次确认
3. 第三道闸：可追溯（装摄像头）

### 3.5.2 四个好习惯

1. 习惯一：安装前，先“安检”：拿到别人写的技能，千万别直接用，先让智能体分析技能的 SKILL.md 和 scripts/目录。
2. 习惯二：第一跑用假数据
3. 习惯三：日常干活，创建专属文件夹（针对所有技能，防止越权）
4. 习惯四：看清确认框，绝不“无脑点‘是’”

### 3.5.3 安全红线清单

| 核心原则     | 红线                                               |
| ------------ | -------------------------------------------------- |
| 默认不喂     | 未脱敏的客户信息、公司财务数据、密码/秘钥          |
| 默认不做     | 删除、覆盖、发布、群发                             |
| 默认不连     | 未经确认的外部 URL（防止拉取恶意代码）             |
| 划定专区     | 只在专属的文件夹内操作，绝不在桌面或系统根目录运行 |
| 首次用新技能 | 用假数据跑一遍，确认安全后再上真数据               |
| 装别人的技能 | 必须查看 SKILL.md + scripts/目录                   |

## 3.6 小结

1. 拿到任务先拆解任务
2. 通过“三问四限”判断技能到底做不做
3. 明确模型与脚本的分工
4. 安装前过安全红线清单
